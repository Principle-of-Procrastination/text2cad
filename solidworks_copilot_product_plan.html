01 · Core Loop

核心产品闭环

LLM 不直接写 API 代码，只产出受控 plan；真正修改模型的是本地 executor。

Prompt

用户在 Task Pane 输入自然语言，并选择面、边、特征或装配体对象。

Context

Add-in 读取当前文档、selection、单位、特征树和关键 metadata。

Feature Plan

planner 输出 JSON plan，包含目标、参数、特征与回滚点。

Preview

用户看到将创建/修改哪些 sketch、dimension、feature，确认后执行。

Executor

本地执行器调 API，处理方向、选择、本地化与失败回滚。

Native Tree

结果保留为原生特征树，可继续手动编辑和重建。

02 · Scope

产品边界

先分清主线与辅助，避免把产品做散。

SolidWorks Add-in主产品形态

能拿到模型上下文、selection、特征树，并直接调 API。

CLI工程底座

适合调试、回归测试、批量任务，不作机械工程师主入口。

独立 Windows App暂不主推

可调 COM API，但上下文与选择体验弱于内嵌 Task Pane。

STEP / Mesh 生成非主线

可作导出或 fallback，但不能作保留设计意图的主数据。

03 · Competitor

CADABRA 研究

官网
执行机制

直接调 API、生成宏、还是混合？决定可靠性与可控性。

原生特征树

是否创建真实 sketch / feature，是否可命名、重建、编辑。

增量编辑

能否修改已有模型，而非只从空白文档生成 demo。

预览与回滚

是否有 preview、确认、checkpoint、undo 或 transaction。

企业包装

账号、SSO、on-prem、数据策略、PDM/PLM、审计与定价。

研究输出：一页竞品拆解表 + 一组实测任务结果，不要只抄官网功能列表。
06 · Bar

MVP 判定标准

成功不是"生成形状"，而是敢改原生模型。

1

原生特征

创建真实 sketch / feature，不生成 Imported body。

2

可继续编辑

特征命名清楚，dimension 可读，能手动改并 rebuild。

3

执行前可确认

用户能看到 plan preview，知道系统要改什么。

4

失败可恢复

失败不破坏模型，有 rollback、日志与明确原因。

04 · Discovery

用户访谈决定 MVP Scope

不要从技术便利性拍板；先确认目标用户最痛、最高频、最愿意交给 AI 的工作流。

访谈对象

机械设计工程师、制造/夹具工程师、CAD 管理员、工程经理，以及已有宏 / DriveWorks / Design Table 的使用者。

访谈问题

哪些操作重复、耗时、易错；哪些能自动执行；哪些必须预览确认；当前自动化方案哪里不满意。

已有零件参数修改

修改厚度、孔径、孔距、圆角、倒角、配置。

频率省时失败风险API 可控付费意愿
标准特征自动化

孔系、Hole Wizard、pattern、紧固件、公司标准特征。

模板化规则清晰依赖选择上下文
出图 / 标注 / BOM

自动生成 drawing package、关键尺寸标注、BOM、导出。

高频当前痛点准确性PDM 依赖
简单零件生成

安装板、支架、夹具等只是候选，不作默认 MVP。

是否真高频明确买家是否强过 CADABRA

07 · Risks

核心风险

最易失败的不是 UI，而是过早泛化与执行可靠性不足。

CADABRA / 官方 LEO 抢占心智

应对：不喊 broad copilot 口号，选一个经访谈验证的高频工作流做深。

SolidWorks API 脆弱性

应对：executor 处理本地化、选择、方向、rebuild、模板差异，建回归测试零件库。

企业数据安全

应对：明确上传范围，支持只传特征树摘要，后续提供 private cloud / on-prem。

Scope 由技术而非用户决定

应对：MVP 场景必须经访谈评分，不把安装板/支架等 demo 直接当产品。
